# QSPI ECO Proposals

Minimal BSV-level changes to fix all documented silicon bugs.
All changes are to `bsv/qspi.bsv` or `bsv/qspi_wrapper.bsv` only.
The compiled Verilog is regenerated by `bsc` — no `.v` edits.

Workaround column refers to `cocotb/cocotb_workaround.md`.

---

## Bug 1 — Remove `cr_en == 0` from `if_abort` guard

**File:** `bsv/qspi.bsv` line 307  
**Retires:** Workaround 2

```diff
- Bool qspi_flush = (cr_abort == 1 || cr_en == 0);
+ Bool qspi_flush = (cr_abort == 1);
```

`transfer_cond` (line 305) already gates every SPI data-transfer rule with `cr_en==1`.
Removing `cr_en==0` from `qspi_flush` stops `if_abort` from preempting AXI reads while
the IP is simply disabled. The IP sits idle without hanging the bus.

## Bug 2 — Remove `cr_en <= 0` from the SLVERR path

**File:** `bsv/qspi.bsv` line 508  
**Retires:** destructive side-effect of Workaround 1

```diff
  wr_rd_resp <= tagged Valid Rd_resp { rsp: axi4_rresp, rdata: 0 };
  axi4_rresp = AXI4_LITE_SLVERR;
  rg_phase <= Idle;
- cr_en <= 0;
  sr_busy <= 0;
  ncs <= 1;
  first_read <= True;
  request_ready = True;
```

A bus decode error should return SLVERR to the master, not self-disable the controller.
All other state is already being cleaned up. The `start_mm_addr=0` in `qspi_wrapper.bsv:8`
is intentional — the XiP window `[0, 0xFFFFFF]` is the physical flash address range in the
SoC; register accesses at `0x02003000+` always exceed `0xFFFFFF` and route correctly to
`access_register()`.

## Bug 3 — Auto-clear `cr_abort` inside `if_abort`

**File:** `bsv/qspi.bsv` lines 756–767  
**Retires:** Workaround 4

```diff
  rule if_abort(qspi_flush);
     rg_phase <= Idle;
     ncs <= 1;
     sr_busy <= 0;
     thres <= False;
     read_true <= False;
     first_read <= False;
     instruction_sent <= False;
     half_cycle_delay <= False;
     fifo.clear();
+    cr_abort <= 0;
  endrule
```

Matches STM32 QUADSPI behaviour: abort fires once, clears all transaction state, then
auto-clears the abort bit. Software detects completion via `SR.BUSY=0` (unchanged).
Requires Bug 1 fix; without it, `cr_en==0` still keeps `qspi_flush` True.

## Bug 4 — Restore STM32 DLR=N-1 convention

**File:** `bsv/qspi.bsv` lines 1302 and 1446  
**Retires:** Workaround 6 (DLR=N convention)

```diff
  // rl_data_read_phase line 1302
- if (count_byte == data_length1) begin
+ if (count_byte == data_length1 + 1) begin

  // rl_data_write_phase line 1446
- if (rg_count_bytes == dlr) begin
+ if (rg_count_bytes == dlr + 1) begin
```

Restores the convention `DLR = N − 1` for N bytes, matching the STM32 reference manual
that RISC-V QSPI drivers are written to.

**Risk — HIGH:** Breaking change for any existing Shakti firmware that already encodes
`DLR = N`. Audit all DLR writes in the driver stack before taping out. After this fix,
revert all `DLR.write(N)` calls in `cocotb/qspi_env.py` to `DLR.write(N-1)`.
Also update Bug 7 ECO threshold expression (see below).

## Bug 5 — Keep TCF high until software clears it via FCR

**File:** `bsv/qspi.bsv` lines 1346 and 1449  
**Retires:** async-watcher hack in `test_21_tcf_interrupt`; "poll BUSY not TCF" note in Workaround 6

```diff
  // rl_data_read_phase line 1346
  sr_tcf <= y;
+ delay_sr_tcf <= y;

  // rl_data_write_phase line 1449
  sr_tcf <= 1;
+ delay_sr_tcf <= 1;
```

`delayed_sr_tcf_signal` (line 618) assigns `sr_tcf <= delay_sr_tcf` on the cycle after
the transfer completes. Currently `delay_sr_tcf=0` (cleared by an earlier phase), so it
overwrites the freshly-set `sr_tcf=1` back to 0 — TCF is observable for zero cycles.
Writing `delay_sr_tcf <= y / 1` in the same cycle feeds the correct value into the delay
register so the next-cycle latch re-asserts TCF. FCR writes clear `sr_tcf` directly
and are unaffected.

## Bug 6 — Suppress spurious TEF when `dcr_fsize = 0` at reset

**File:** `bsv/qspi.bsv` lines 856 and 858  
**Retires:** "ignore SR.TEF" note from Bug 6 workaround

```diff
  rule set_error_signal;
     Bit#(32) actual_address = 1 << dcr_fsize;
-    if (wr_address_written && ar > actual_address && (ccr_fmode=='b00 || ccr_fmode=='b01))
+    if (wr_address_written && dcr_fsize != 0 && ar > actual_address && (ccr_fmode=='b00 || ccr_fmode=='b01))
        sr_tef <= 1;
-    else if (wr_address_written && ar+dlr > actual_address && (ccr_fmode=='b00 || ccr_fmode=='b01))
+    else if (wr_address_written && dcr_fsize != 0 && ar+dlr > actual_address && (ccr_fmode=='b00 || ccr_fmode=='b01))
        sr_tef <= 1;
     else if (wr_address_written)
        sr_tef <= 0;
  endrule
```

When `dcr_fsize=0` the flash size has not been programmed; `1<<0=1` is not a meaningful
limit. Guard suppresses TEF until fsize is configured, matching production behaviour.

## Bug 7 — Start indirect write when FIFO has `min(DLR, 16)` bytes

**File:** `bsv/qspi.bsv` line 839  
**Retires:** Workaround 8 (FIFO-padding loop for indirect writes < 16 bytes)

```diff
  rule rl_data_wait(sr_busy==0 && rg_phase==DataWait_phase && cr_abort==0 && cr_en==1);
-    if (fifo.count >= 16) begin
+    if (fifo.count >= (dlr < 16 ? unpack(truncate(dlr + 1)) : 5'd16)) begin
```

For `dlr >= 16` the threshold is 16 — identical to current behaviour.
For `dlr < 16` the SPI engine starts as soon as all `dlr+1` data bytes are in the FIFO.

**Bug 4 interaction:**
- If Bug 4 IS fixed (DLR=N−1): threshold `dlr+1` is correct — "N−1 bytes written → N−1+1 = N bytes in FIFO".
- If Bug 4 is NOT fixed (DLR=N): change `dlr+1` to `dlr`.

## Bug 8 — Fix DR byte ordering for awsize≥2 and arsize≥2

**File:** `bsv/qspi.bsv` lines 441–448 (write) and 587–591 (read)  
**Retires:** Workaround 8 (`arsize=2'd0`/`awsize=2'd0` hardcoding in `tb_qspi.v`)

### Write path (awsize ≥ 2), lines 441–448

```diff
  else begin
     dr <= wdata[31:0];
     Vector#(4,Bit#(8)) temp = newVector();
-    temp[0] = wdata[31:24];
-    temp[1] = wdata[23:16];
-    temp[2] = wdata[15:8];
-    temp[3] = wdata[7:0];
+    temp[0] = wdata[7:0];
+    temp[1] = wdata[15:8];
+    temp[2] = wdata[23:16];
+    temp[3] = wdata[31:24];
     if (fifo.enqReadyN(4))
     fifo.enq(4,temp);
  end
```

### Read path (arsize ≥ 2), lines 587–591

```diff
  else begin
     if (fifo.deqReadyN(4)) begin
-        let temp = {fifo.first[0], fifo.first[1], fifo.first[2], fifo.first[3]};
+        let temp = {fifo.first[3], fifo.first[2], fifo.first[1], fifo.first[0]};
         reg1 = duplicate(temp);
         fifo.deq(4);
     end
  end
```

Makes the 32-bit path little-endian-first, consistent with the awsize=0/arsize=0 byte
path that production firmware uses. awsize=1 at lines 434–435 has the same inversion
(`temp[0]=wdata[15:8], temp[1]=wdata[7:0]`); fix that too if 16-bit DR writes are
expected:

```diff
- temp[0] = wdata[15:8];
- temp[1] = wdata[7:0];
+ temp[0] = wdata[7:0];
+ temp[1] = wdata[15:8];
```

## Bug 9 — Guard RX FIFO enqueue in `rl_data_read_phase`

**File:** `bsv/qspi.bsv` lines 1264, 1279, 1292  
**Retires:** Workaround 9 (16-byte chunk splitting)

```diff
  // single-line mode — line 1264
- if (!first_read)
+ if (!first_read && fifo.enqReadyN(1))
  fifo.enq(1, temp);

  // dual-line mode — line 1279
- if (!first_read)
+ if (!first_read && fifo.enqReadyN(1))
  fifo.enq(1, temp);

  // quad-line mode — line 1292
- if (!first_read)
+ if (!first_read && fifo.enqReadyN(1))
  fifo.enq(1, temp);
```

Mirrors the TX write path (lines 428–429) which already has this guard. When the FIFO
is full the incoming byte is dropped and the SPI engine stalls (SCK continues toggling
but data is lost) until software drains DR — natural backpressure. `count_byte`
still increments so the DLR terminal condition fires at the right time.

**Note:** With this fix, reads longer than 16 bytes require software to drain the FIFO
mid-burst (poll FTF / use interrupt). The 16-byte chunk-split workaround in `qspi_env.py`
can be removed; callers must instead drain DR promptly for large reads.

## Possible Bug 10 — Missing synchronizer on `rg_input` (MISO / DQ)

**File:** `bsv/qspi.bsv` line 162

```bsv
Wire#(Bit#(4)) rg_input <- mkDWire(0);   // no synchronizer
```

The `io_i` method (line 1502–1505) directly assigns the external `io_in` pins to the
`mkDWire` `rg_input` with no two-flop synchronizer. In the SoC the MISO/DQ lines are
asynchronous with respect to the system clock, creating a metastability risk.

An earlier version of the IP reportedly used `mkSyncBit` or a similar primitive here.
If the synchronizer was removed before tapeout this needs to be confirmed and restored:

```diff
- Wire#(Bit#(4)) rg_input <- mkDWire(0);
+ SyncBitIfc#(Bit#(4)) rg_input_sync <- mkSyncBits(0, slow_clk, slow_rst, clk, rst);
  // then in io_i method:
- rg_input <= io_in;
+ rg_input_sync.send(io_in);
  // and in rules reading rg_input:
- ... rg_input ...
+ ... rg_input_sync.read ...
```

**Action required:** Verify whether a synchronizer was present in the pre-tapeout source.
If it was removed, this is a functional silicon bug for any board operating above a few
MHz SPI clock. Not included in the numbered bug list until confirmed.
